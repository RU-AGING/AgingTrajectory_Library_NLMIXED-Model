What is GBTM?

🧭

Group-Based Trajectory Modeling

Nagin (1999) · Finite Mixture Model · Maximum Likelihood via PROC NLMIXED

Group-based trajectory modeling (GBTM) is a finite mixture model that stratifies the sample into a small number of latent groups, gives each group its own polynomial time trend, and jointly estimates the class shapes and mixing weights by maximum likelihood (Nagin 1999). Rather than estimating a single average trajectory for everyone, which typically fails to capture the heterogeneity visible in longitudinal data: GBTM partitions the population into C latent classes, each characterized by its own trajectory shape estimated as a polynomial function of time. These classes represent patterns rather than a typology: they are longitudinal strata, not sharply bounded natural categories.

Each subject's observed longitudinal sequence contributes to the likelihood through two components: the class-membership probability πj (multinomial, constrained to sum to 1, with a clear interpretation as the expected class proportion) and the class-conditional distribution of the observed data given membership. Parameters are estimated jointly by maximum likelihood using the SAS internal optimizers of PROC NLMIXED (dual quasi-Newton and Newton-Raphson). The polynomial order can be linear, quadratic, cubic, or any order, and can differ between classes.

P(Y) = Σ_{j=1..C} π_j · Π_{k=1..K} Π_{t=1..T} p(Y_t^k | X = j) where X is the latent class (1..C), K ∈ {1,2} outcomes, T time points, and p(Y_t^k | X=j) is the class-conditional density (censored normal or cumulative ordinal probit in this release)

Assumptions. (1) Conditional independence over time: given class membership, responses at different time points are independent. (2) Conditional independence across outcomes: given class membership, there is no association between the two outcomes. Skipping missing time points in the likelihood is valid under ignorable missingness (missing at random); data missing not at random would require an explicit missingness model (Dodge et al. 2008).

The current release implements two outcome families: bounded ordinal scales (class-specific cumulative probit with exp-incremented thresholds) and continuous measures with bounded support (class-specific censored normal, with built-in estimation of joint two-outcome trajectories following the multi-trajectory framework of Nagin et al. 2018). Zero-inflated count prototypes (ZIP and ZINB) are in the repository and undergoing quality assurance: see Planned Extensions below.

Why a Native Implementation?

The well-known SAS procedure PROC TRAJ implements GBTM but relies on an externally compiled C dynamic-link library, which prevents its use in environments that restrict externally compiled code, most importantly the CMS Virtual Research Data Center (VRDC), where an increasing amount of Medicare and Medicaid research is conducted, as well as VA data enclaves and other secure custodial platforms. Traj2 expresses the GBTM marginal likelihood directly in PROC NLMIXED (included in SAS/STAT in every standard SAS distribution) with no compiled components, so the macros run as-is in the VRDC and with any version of SAS 9.4 and newer. Because the likelihoods are written in SAS macro language, analysts can read, edit, and extend them, adding outcome types the compiled procedure does not support, and diagnosing convergence problems that a black-box estimator would hide.

Validation Highlights

🎯

Agreement with PROC TRAJ

On identical simulated data (N = 10,000, three-class quadratic censored-normal), the single-outcome fit matches PROC TRAJ with a maximum absolute coefficient difference of 1.0 × 10⁻⁴ across all nine coefficients. The joint two-outcome (multi-trajectory) fit matches to ≈ 5 × 10⁻⁴ across all 22 parameters; class proportions agree to three decimals and −2 log L differs by about 0.2.

Numerical-tolerance match
🎲

Monte Carlo Bias & Coverage

Over 200 independent replications from the three-class censored-normal DGP (N = 2,500 each), the joint two-outcome model shows a maximum absolute bias of ≈ 0.011 on a 0–100 scale, and empirical 95% CI coverage from 0.930 to 0.975 (mean 0.953), consistent with nominal coverage. A Monte Carlo scaffold with the design and seed is distributed with the package.

Near-nominal coverage
📐

Ordinal-Probit Recovery

Over 200 replications (N = 2,500, T = 12, true C = 3), worst-class trajectory RMSE is 0.029 on the 0–3 scale; class assignment agrees with truth 91.0% of the time (vs. a 45% majority-class baseline); and BIC selected the true C = 3 in 100% of replications. All 200 replications returned fully estimable standard errors for every free parameter.

Shape · Class · C recovered
🔁

Cross-Implementation Check (lcmm)

Because PROC TRAJ lacks a cumulative-probit family, the ordinal model was checked against the R package lcmm on a matched design: fitted E[Y | class] trajectories agree to a maximum absolute difference of 0.012 (mean 0.003) across 36 class-by-time points, class proportions differ by at most 1 percentage point, and −2 log L is essentially identical (68,517 vs. 68,520).

Real-Data Application: Cognitive Trajectories in the Last Five Years of Life

🧠

2019 Medicare Decedent Cohort · CMS VRDC

Lin, April-Sanders, Zafar & Jarrín (2026), manuscript submitted for publication

Using the ordinal-probit pipeline (%ordprob_mix_fit_one), cognitive function during the final five years of life was classified in a 10% random sample (N = 181,463) of Medicare beneficiaries older than 50 years who died in 2019. Assessments were quarterly over T = 20 quarters, in four categories of cognitive impairment (none, memory loss, mild, moderate-to-severe/dementia), with each class trajectory fitted as a quadratic function of time on the cumulative-probit scale.

Models with 2–11 classes were fit; by BIC, a ten-class solution was preferred. Recovered trajectories ranged from a large group with near-stable cognition (35.9% of the sample) to a group severely impaired for the entire duration (12.1%). Reproducibility was confirmed by refitting over several different 10% random samples, each returning the same ten-class solution. The entire analysis was carried out inside the restricted CMS VRDC, where PROC TRAJ's compiled binary is unavailable, precisely the setting that motivated Traj2, and demonstrates scalability: 181,463 subjects, 20 time points, ten latent classes (59 parameters), fit in ~53 minutes wall-clock.

Censored-Normal GBTM: Single & Joint Two-Outcome

Authors: Weiyi Xia, Haiqun Lin, Anum Zafar

For continuous outcomes with a lower and/or upper bound, common in clinical biomarker data that cannot be measured below the limit of quantitation, and in bounded count data approximated as continuous, Traj2 uses a censored-normal (Tobit) density. With the class-specific latent mean μjt = βj0 + βj1t + ... + βjdtd, and φ, Φ the standard normal density and CDF, the class-conditional contribution at time t is:

p(Y_t | X = j) = σ_j⁻¹ φ((Y_t − μ_jt)/σ_j) if L < Y_t < U (observed exactly) Φ((L − μ_jt)/σ_j) if Y_t = L (lower-censored) 1 − Φ((U − μ_jt)/σ_j) if Y_t = U (upper-censored)

The density over the interior is not renormalized over [L, U]; the bounds carry positive probability mass, as opposed to a truncated normal, which would renormalize the interior density and give the bounds no mass. This matches the CNORM family in PROC TRAJ. Censoring is determined by an exact equality of Yt to a bound: the user-supplied cap vector must exactly match the ceiling of the data. If a cap value is lower than any actual observed outcome, cases at that cap are treated as interior and contribute no censored component, always validate the cap values against the maximum of the sample.

The joint two-outcome model links two co-evolving continuous measures through a shared latent-class structure, following the multi-trajectory framework of Nagin et al. (2018): the latent class is a subject-level attribute, and each class has one trajectory per outcome. The model assumes conditional independence over time and across the two outcomes, given class membership.

Why warm starts matter. The joint optimization is more prone to convergence failure because of its sensitivity to starting values. Traj2 therefore uses a warm-start pipeline: the two outcomes are first fit separately with %nlmixed_1, their parameter estimates are stacked (dropping the mixing intercepts) into work.nlm_2y_starting, and %nlmixed_MultiTraj reads these per-parameter starting values via the / data= clause on the PARMS statement, while %starting_value_alpha provides neutral starting values for the mixing intercepts.

Joint Two-Outcome Warm-Start Pipeline

SAS · Warm-start pipeline (fit separately → stack → joint fit)
/* 1. Fit each outcome separately */
%nlmixed_1(T=12, LC=3, Y=1,
  starting=%starting_value_alpha(class=3)
           %starting_value_beta_sigma(class=3, outcome=1,
             order=2, equal_sigma=T),
  output=nlm_fix_T1C3, order=2, equal_sigma=T);

%nlmixed_1(T=12, LC=3, Y=2,
  starting=%starting_value_alpha(class=3)
           %starting_value_beta_sigma(class=3, outcome=2,
             order=2, equal_sigma=T),
  output=nlm_fix_T2C3, order=2, equal_sigma=T);

/* 2. Stack the single-outcome estimates, dropping the mixing
      intercepts, to seed the joint fit */
data work.nlm_2y_starting;
  set nlm_fix_T1C3 nlm_fix_T2C3;
  if parameter =: 'alpha' then delete;
run;

/* 3. Fit the joint model, reading starting values from the dataset */
%nlmixed_MultiTraj(T=12, LC=3,
  starting=%starting_value_alpha(class=3) / data=work.nlm_2y_starting,
  output=nlm_fix_T1_T2C3, order=2, equal_sigma=T);

%plot_prep(T=12, LC=3, result=nlm_fix_T1_T2C3,
  order=2, equal_sigma=T);
💡
Run order matters. Always fit %nlmixed_1(Y=1) and %nlmixed_1(Y=2) first, then build work.nlm_2y_starting with the DATA step above (dropping the alpha rows) before calling %nlmixed_MultiTraj. The / data= clause on the PARMS statement reads per-parameter starting values from the dataset, while %starting_value_alpha provides neutral starting values for the mixing intercepts. Starting the joint model cold significantly increases the risk of non-convergence.

Ordinal-Probit Latent-Class Trajectory Model

Authors: Haiqun Lin, Anum Zafar

For bounded, ordered scales such as activities of daily living (ADL), frailty indices, and cognitive screening scores, Traj2 uses a class-specific cumulative probit. For an ordinal outcome with M ordered categories taking value m in {0, 1, ..., M−1}, each class j has a latent index that is a polynomial in time, together with class-specific thresholds:

η_jt = β_j0 + β_j1·t + ... + β_jd·t^d [class-specific latent index] P(Y_t = m | X = j) = Φ(τ_j,m+1 − η_jt) − Φ(τ_j,m − η_jt) with τ_j,0 = −∞, τ_j,M = +∞, and Φ the standard normal CDF. Thresholds (built from exponential increments): τ_j,1 = 0 [FIXED] τ_j,m = τ_j,m−1 + exp(i_j,m), m = 2, ..., M−1

Building the thresholds from exponential increments ensures the ordering τj,m > τj,m−1 in a way that standard nonlinear optimizers can handle: exp(·) > 0 guarantees strict monotonicity without constrained optimization.

Identification. The lowest threshold is fixed at τj,1 = 0 and the class intercept βj0 is estimated freely; this identifies the location of each class's latent index, so the free threshold parameters are only the increments ij,2, ..., ij,M−1. Users wishing to provide their own starting values need only enter the increments and need not provide a value for the first threshold. The Class A mixing logit is likewise fixed to zero for identification, never include it in start_values. Fixing τj,1 = 0 removes the location aliasing between βj0 and the first threshold, so the model is fully identified and standard errors are estimable: in validation, all 200 Monte Carlo replications returned non-missing, positive standard errors for every free parameter.

Zero-Inflated Poisson (ZIP) Trajectories

Author: Anum Zafar · Prototype under QA · Last updated 02/09/2026

The ZIP model addresses a fundamental challenge in healthcare count data: the presence of far more zeros than a standard Poisson distribution can accommodate. In Medicare claims data, many beneficiaries have zero utilization in a given quarter, not because they were never at risk, but because the underlying care process generates both structural zeros (individuals who would not use the service regardless) and sampling zeros (individuals who happen not to use it in that period). ZIP separates these two processes explicitly.

For each class k and time point t, ZIP models the outcome Y as a two-component mixture:

P(Yᵢₜ = 0 | class k) = pₖ(t) + (1 − pₖ(t)) · exp(−μₖ(t)) P(Yᵢₜ = y | class k) = (1 − pₖ(t)) · [exp(−μₖ(t)) · μₖ(t)ʸ / y!] for y > 0 where μₖ(t) = exp(β₀ₖ + β₁ₖ·t + β₂ₖ·t²) [mean count trajectory, log-link] pₖ(t) = logistic(γ₀ₖ + γ₁ₖ·t) [zero-inflation logit]

The mean count trajectory μₖ(t) captures how the expected frequency of events (e.g., ED visits per quarter) evolves over 12 quarters within each class, fitted as a polynomial of degree order= on the log scale. The zero-inflation parameter pₖ(t) captures the proportion of structural non-users in that class, a time-varying probability that can itself trend up or down.

Mixing weights π₁…πK (from softmax on α parameters) determine the relative size of each class in the population. Model selection uses BIC, prefer the K that gives the lowest BIC while keeping all class sizes above ~5% and posterior assignment probabilities above 0.7.

ED visits · quarterly Outpatient encounters Skilled nursing facility days Prescription fills Any CMS count outcome with excess zeros

Interpreting Results

Mixing weights (α)

alpha0_* are softmax logits; Class A is reference (α₀_A = 0 implicit). The estimated share of class k is exp(αₖ) / Σexp(αⱼ). Larger alpha = larger class proportion in the population.

Mean curve (β)

μₖ(t) = exp(β₀ₖ + β₁ₖ·t + β₂ₖ·t²). Positive β₁ = upward trend; negative β₂ = concave-down peak. Estimates and SEs in pe_zip. AIC/BIC for K selection in fit_zip.

Zero-inflation (γ)

pₖ(t) = logistic(γ₀ₖ + γ₁ₖ·t). Higher γ₀ = more structural non-users in that class. If γ₁ ≠ 0, structural zeros are trending over time. Observed mean = (1 − pₖ) × μₖ.

Zero-Inflated Negative Binomial (ZINB) Trajectories

Author: Anum Zafar · Prototype under QA · Last updated 02/09/2026

The ZINB model extends ZIP by relaxing the Poisson assumption that variance equals the mean. In Medicare data (where a small fraction of high-need beneficiaries accounts for a disproportionate share of utilization), count variance routinely far exceeds the mean even within a given class. This extra-Poisson variability is called overdispersion, and ignoring it inflates standard errors and distorts trajectory shapes.

ZINB achieves this by replacing the Poisson component with a Negative Binomial (NB2) distribution. NB2 is obtained by allowing the Poisson rate λ to itself be Gamma-distributed: λ ~ Gamma(k, μ/k), then Y | λ ~ Poisson(λ). The resulting marginal distribution has mean μ and variance μ + μ²/k, where k is the class-specific dispersion parameter. As k → ∞, variance → mean and ZINB collapses to ZIP. Small k indicates severe overdispersion.

P(Yᵢₜ = 0 | k) = pₖ(t) + (1 − pₖ(t)) · (kₖ / (kₖ + μₖ(t)))^kₖ P(Yᵢₜ = y | k) = (1 − pₖ(t)) · NB(y; μₖ(t), kₖ) for y > 0 kₖ = exp(logk_CLASS) [log scale ensures positivity] Var(Y | k) = μₖ + μₖ² / kₖ [vs. Var = μ for Poisson]

The macro implements a numerically stable log-sum-exp formulation for the zero-mass probability to prevent floating-point underflow, which is critical for high-utilization classes where both pₖ and the NB zero probability can be very small. Warm starts for logk_* should begin at 0 (k = 1, moderate dispersion) and can be adjusted if convergence is slow.

When to prefer ZINB over ZIP: Run ZIP first. If NLMIXED log shows convergence difficulties, if residual plots show systematic over-prediction of low counts, or if likelihood ratio test of ZIP vs ZINB (difference in log-likelihoods) is large, switch to ZINB. The extra parameter per class (logk_*) costs degrees of freedom but is often worth it for CMS claims data.

High-utilization subgroups in Medicare Inpatient admissions (heavy tail) Post-acute care days Any count where Var >> Mean

Docker: Run Traj2 Without Local Setup

A companion Docker container with the full macro library, simulators, and entrypoint scripts. Pull, mount your licensed SAS installation, and run any released model with a single command.

Image: chaolab/gbtm-macros:1.0.0 Requires: user-supplied licensed SAS 9.4+ mounted at /opt/sas
⚠️
SAS license required. The image does not bundle SAS, since that would violate SAS Institute's licensing terms. You mount your own licensed SAS 9.4+ installation read-only at /opt/sas when you run the container.

Pull the Image

Shell
docker pull chaolab/gbtm-macros:1.0.0

Validate Your Setup

Confirm the image can find and run your SAS installation:

Shell
docker run --rm \
  -v /path/to/your/sas:/opt/sas:ro \
  chaolab/gbtm-macros:1.0.0 validate

Run a Model

Fit a 4-class ordinal-probit model on data in ./data, with outputs written to ./output:

Shell
docker run --rm \
  -v /path/to/your/sas:/opt/sas:ro \
  -v "$PWD/data":/opt/gbtm/data \
  -v "$PWD/output":/opt/gbtm/output \
  chaolab/gbtm-macros:1.0.0 \
  run --model=ordinal --nclass=4 --order=2

Released models: ordinal, continuous. Prototype models (zip, zinb) are included for QA testing only.

Or Use Docker Compose

Copy docker-compose.yml from the repo, set SAS_HOME in a .env file, then:

Shell
GBTM_MODEL=ordinal GBTM_NCLASS=4 docker compose run --rm gbtm

What's Inside the Image

Pre-loaded macros

Both released model families (Ordinal-Probit and Censored-Normal continuous), the ZIP/ZINB prototypes, simulators, and plotting utilities at /opt/gbtm/macros.

Entrypoint commands

help, validate, list, run, and shell; invoke with docker run … chaolab/gbtm-macros:1.0.0 <command>.

Bring your own SAS

Mount a licensed SAS 9.4+ installation at /opt/sas. The entrypoint locates the sas binary under standard install paths automatically.

Reproducible tags

Image tags match the Git release tags one-to-one; :latest tracks the default branch.

ℹ️
Linux/amd64 only. The image is built for x86_64. ARM (Apple Silicon) users need to invoke Docker with --platform linux/amd64; emulation works but is slower.
