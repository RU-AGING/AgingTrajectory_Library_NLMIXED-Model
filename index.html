What is GBTM?

🧭

Group-Based Trajectory Modeling

Nagin & Land (1993) · Finite Mixture Model · Maximum Likelihood via PROC NLMIXED

Group-Based Trajectory Modeling (GBTM) is a finite mixture model designed to identify subgroups of individuals in a population who follow similar patterns over time. Rather than estimating a single average trajectory for everyone, which can mask meaningful heterogeneity, GBTM partitions the population into K latent classes, each characterized by its own trajectory shape (intercept, slope, curvature) estimated as a polynomial function of time.

The core idea is that the population distribution of trajectories is approximated by a finite mixture of K groups. This is not a claim that nature has produced exactly K discrete types; it is a flexible, data-driven device for capturing the dominant features of individual-level heterogeneity. The latent groups should be thought of as longitudinal strata, clusters of individuals following approximately the same developmental course, rather than as sharply bounded categories.

Each individual's observed longitudinal sequence contributes to the likelihood through two components: the probability of group membership (modeled via a generalized logit / softmax) and the probability of the observed data given group membership (modeled by a distribution appropriate to the outcome type). Parameters are estimated jointly by maximum likelihood. Model selection, choosing K, relies on BIC, AIC, average posterior probability, and clinical interpretability of the resulting trajectories.

The library implements GBTM for outcome types common in CMS Medicare/Medicaid data, including bounded ordered outcomes (Ordinal-Probit) and bounded continuous outcomes (truncated normal, with single- and two-outcome variants). Count-outcome extensions (ZIP and ZINB) build on the same finite-mixture framework and shared data contract.

L(θ) = ∏ᵢ Σⱼ πⱼ · ∏ₜ P(yᵢₜ | group j, θⱼ) where πⱼ = exp(αⱼ) / Σⱼ exp(αⱼ) [softmax mixing weights] and P(yᵢₜ | j) depends on the outcome distribution (Ordinal-Probit, Truncated Normal, ZIP, or ZINB)

Models at a Glance

🔢

Ordinal-Probit

For bounded ordered-categorical outcomes (e.g., functional status scales, frailty scores, severity grades 0–4). Uses cumulative probit with exp-incremented thresholds that guarantee strict category ordering while allowing each class to follow its own polynomial mean trajectory.

Ordered Categories
📉

Continuous (2-outcome truncated normal)

For continuous outcomes on bounded support, with optional one- or two-sided bounds that may vary across time. Jointly models two outcomes — for example, home-health visits and inpatient days — across K classes, using single-outcome warm starts to stabilize the joint optimization.

Bounded · Joint
📊

ZIP Poisson (extension)

For integer count outcomes (e.g., quarterly ED visits, hospitalizations) with excess zeros. Builds on the same finite-mixture framework as the released models, replacing the class-conditional density with a zero-inflated Poisson while preserving the shared data contract and softmax mixing weights.

Count · Zero-Inflated
📈

ZINB (extension)

Extends ZIP by replacing the Poisson component with a Negative Binomial (NB2) distribution to accommodate overdispersion (Var = μ + μ²/k). Appropriate when count variance substantially exceeds the mean even after accounting for structural zeros, common in Medicare claims data for high-utilization populations.

Count · Overdispersed

Ordinal-Probit Latent-Class Trajectory Model

Authors: Haiqun Lin, Anum Zafar

For bounded ordered categorical outcomes such as activities of daily living (ADL) grades, frailty indices, pain severity, and cognitive screening scores, Traj2 assumes a latent continuous variable Y* whose class-conditional mean is a polynomial function of time. The observed category probabilities are differences of standard-normal cumulative distribution functions evaluated at class-specific thresholds:

η_k(t) = β_k0 + β_k1·t + β_k2·t² + ... [class-specific mean of Y*] P(Y = j | class k, t) = Φ(τ_k,j − η_k(t)) − Φ(τ_k,j-1 − η_k(t)) Thresholds (parameterized to ensure strict ordering): τ_k,1 = i_k1 τ_k,j = τ_k,j-1 + exp(i_k,j) for j ≥ 2

The use of exp-incremented thresholds is a key design choice. Parameterizing threshold gaps as exp(i_k,j) rather than raw differences automatically guarantees τ_k,1 < τ_k,2 < τ_k,3 < ... without requiring constrained optimization, which substantially improves numerical stability in PROC NLMIXED relative to a raw-threshold parameterization.

Identifiability. In the cumulative probit, the location of the latent scale is determined jointly by the polynomial intercept β_k0 and the first threshold i_k,1; only one of them can be identified. PROC NLMIXED will typically report a Moore–Penrose generalized inverse and missing standard errors for one of these parameters. This is expected and does not affect estimated category probabilities, posterior assignments, or trajectory shapes. Clean standard errors are obtained by fixing all β_k0 = 0, or alternatively all i_k,1 = 0, in starting values. The library never fits α_A (Class A reference logit) for the same reason — it is pinned to zero internally.

ADL / IADL functional status Frailty index grades Cognitive screening scores Pain severity scales Any 0–M bounded ordinal outcome

Two Continuous Outcomes — Truncated-Normal GBTM

Authors: Weiyi Xia, Haiqun Lin, Anum Zafar

For continuous outcomes restricted to a bounded support — for example, utilization measures with a known per-period maximum, or biomarker readings with floor or ceiling effects — standard linear GBTM places probability mass outside the supported range and biases trajectory estimates near the boundaries. The truncated-normal distribution addresses this by restricting the Gaussian density to the user-specified support [L_t, U_t] and renormalizing accordingly. Bounds may be one-sided (lower only or upper only) or two-sided, and may vary across time points.

μ_k(t) = β_k0 + β_k1·t + β_k2·t² + ... [class-specific mean] Truncated-normal density at time t, on support [L_t, U_t]: f(y | k, t) = (1/σ_k) · φ((y − μ_k(t)) / σ_k) ────────────────────────────────────────── Φ((U_t − μ_k(t))/σ_k) − Φ((L_t − μ_k(t))/σ_k) for L_t ≤ y ≤ U_t.

The bounds (L_t, U_t) must be stored as actual data columns in BASE_FILE_SRS — not just as macro variable values. The %build_base_file_srs macro handles this, which is easy to overlook and causes silent errors if done manually.

The library extends single-outcome estimation to joint two-outcome models in which two co-evolving continuous measures share a common latent-class structure. Subjects in Class A have one trajectory for Outcome 1 and a separate trajectory for Outcome 2, subjects in Class B have different trajectories for each, and so on. This reveals how two correlated processes (for example, two complementary care utilization measures) co-evolve within each subpopulation. The joint formulation assumes conditional independence between the two outcomes given class and time.

Why warm starts matter. The joint K-class parameter space is substantially larger than either single-outcome space. Cold starts from zero frequently land in local maxima or fail to converge. The library therefore enforces a warm-start workflow: %nlmixed_1(Y=1) and %nlmixed_1(Y=2) are run first to obtain single-outcome estimates, which are automatically written to work.nlm_2y_starting and read by %nlmixed_MultiTraj as starting values for the joint optimization.

Home health + inpatient (joint) HCC scores (bounded range) Cost trajectories with ceiling Two correlated bounded outcomes

Zero-Inflated Poisson (ZIP) Trajectories

Author: Anum Zafar · Last updated 02/09/2026

The ZIP model addresses a fundamental challenge in healthcare count data: the presence of far more zeros than a standard Poisson distribution can accommodate. In Medicare claims data, many beneficiaries have zero utilization in a given quarter, not because they were never at risk, but because the underlying care process generates both structural zeros (individuals who would not use the service regardless) and sampling zeros (individuals who happen not to use it in that period). ZIP separates these two processes explicitly.

For each class k and time point t, ZIP models the outcome Y as a two-component mixture:

P(Yᵢₜ = 0 | class k) = pₖ(t) + (1 − pₖ(t)) · exp(−μₖ(t)) P(Yᵢₜ = y | class k) = (1 − pₖ(t)) · [exp(−μₖ(t)) · μₖ(t)ʸ / y!] for y > 0 where μₖ(t) = exp(β₀ₖ + β₁ₖ·t + β₂ₖ·t²) [mean count trajectory, log-link] pₖ(t) = logistic(γ₀ₖ + γ₁ₖ·t) [zero-inflation logit]

The mean count trajectory μₖ(t) captures how the expected frequency of events (e.g., ED visits per quarter) evolves over 12 quarters within each class, fitted as a polynomial of degree order= on the log scale. The zero-inflation parameter pₖ(t) captures the proportion of structural non-users in that class, a time-varying probability that can itself trend up or down.

Mixing weights π₁…πK (from softmax on α parameters) determine the relative size of each class in the population. Model selection uses BIC, prefer the K that gives the lowest BIC while keeping all class sizes above ~5% and posterior assignment probabilities above 0.7.

Interpreting Results

Mixing weights (α)

alpha0_* are softmax logits; Class A is reference (α₀_A = 0 implicit). The estimated share of class k is exp(αₖ) / Σexp(αⱼ). Larger alpha = larger class proportion in the population.

Mean curve (β)

μₖ(t) = exp(β₀ₖ + β₁ₖ·t + β₂ₖ·t²). Positive β₁ = upward trend; negative β₂ = concave-down peak. Estimates and SEs in pe_zip. AIC/BIC for K selection in fit_zip.

Zero-inflation (γ)

pₖ(t) = logistic(γ₀ₖ + γ₁ₖ·t). Higher γ₀ = more structural non-users in that class. If γ₁ ≠ 0, structural zeros are trending over time. Observed mean = (1 − pₖ) × μₖ.

Zero-Inflated Negative Binomial (ZINB) Trajectories

Author: Anum Zafar · Last updated 02/09/2026

The ZINB model extends ZIP by relaxing the Poisson assumption that variance equals the mean. In Medicare data — where a small fraction of high-need beneficiaries accounts for a disproportionate share of utilization — count variance routinely far exceeds the mean even within a given class. This extra-Poisson variability is called overdispersion, and ignoring it inflates standard errors and distorts trajectory shapes.

ZINB achieves this by replacing the Poisson component with a Negative Binomial (NB2) distribution. NB2 is obtained by allowing the Poisson rate λ to itself be Gamma-distributed: λ ~ Gamma(k, μ/k), then Y | λ ~ Poisson(λ). The resulting marginal distribution has mean μ and variance μ + μ²/k, where k is the class-specific dispersion parameter. As k → ∞, variance → mean and ZINB collapses to ZIP. Small k indicates severe overdispersion.

P(Yᵢₜ = 0 | k) = pₖ(t) + (1 − pₖ(t)) · (kₖ / (kₖ + μₖ(t)))^kₖ P(Yᵢₜ = y | k) = (1 − pₖ(t)) · NB(y; μₖ(t), kₖ) for y > 0 kₖ = exp(logk_CLASS) [log scale ensures positivity] Var(Y | k) = μₖ + μₖ² / kₖ [vs. Var = μ for Poisson]

The macro implements a numerically stable log-sum-exp formulation for the zero-mass probability to prevent floating-point underflow, which is critical for high-utilization classes where both pₖ and the NB zero probability can be very small. Warm starts for logk_* should begin at 0 (k = 1, moderate dispersion) and can be adjusted if convergence is slow.

When to prefer ZINB over ZIP: Run ZIP first. If NLMIXED log shows convergence difficulties, if residual plots show systematic over-prediction of low counts, or if likelihood ratio test of ZIP vs ZINB (difference in log-likelihoods) is large, switch to ZINB. The extra parameter per class (logk_*) costs degrees of freedom but is often worth it for CMS claims data.

High-utilization subgroups in Medicare Inpatient admissions (heavy tail) Post-acute care days Any count where Var >> Mean

Docker — Run Traj2 Without Local Setup

A prebuilt Docker image with the full macro library, simulators, and entrypoint scripts is published to GitHub Container Registry. Pull, mount your licensed SAS installation, and run any model with a single command.

Image: ghcr.io/ru-aging/traj2:1.0.0 Requires: Licensed SAS 9.4+ Size: ~80 MB (excluding SAS)
⚠️
SAS license required.The image does not bundle SAS — that would violate SAS Institute's licensing terms. You mount your own licensed SAS 9.4+ installation read-only at /opt/sas when you run the container.

Pull the Image

Shell
docker pull ghcr.io/ru-aging/traj2:1.0.0

Validate Your Setup

Confirm the image can find and run your SAS installation:

Shell
docker run --rm \
  -v /path/to/your/sas:/opt/sas:ro \
  ghcr.io/ru-aging/traj2:1.0.0 validate

Run a Model

Fit a 4-class ordinal-probit model on data in ./data, with outputs written to ./output:

Shell
docker run --rm \
  -v /path/to/your/sas:/opt/sas:ro \
  -v "$PWD/data":/opt/gbtm/data \
  -v "$PWD/output":/opt/gbtm/output \
  ghcr.io/ru-aging/traj2:1.0.0 \
  run --model=ordinal --nclass=4 --order=2

Available models: ordinal, continuous, zip, zinb.

Or Use Docker Compose

Copy docker-compose.yml from the repo, set SAS_HOME in a .env file, then:

Shell
GBTM_MODEL=ordinal GBTM_NCLASS=4 docker compose run --rm gbtm

All four model families (Ordinal-Probit, Truncated-Normal continuous, ZIP, ZINB), simulators, and plotting utilities at /opt/gbtm/macros.

help, validate, list, run, and shell — invoke with docker run … ghcr.io/ru-aging/traj2:1.0.0 <command>.

Bring your own SAS

Mount a licensed SAS 9.4+ installation at /opt/sas. The entrypoint locates the sas binary under standard install paths automatically.

Reproducible tags

Image tags match the Git release tags one-to-one. v1.0.0 on GitHub builds :1.0.0 on GHCR. :latest tracks the default branch.

ℹ️
Linux/amd64 only.The image is built for x86_64. ARM (Apple Silicon) users need to invoke Docker with --platform linux/amd64; emulation works but is slower.

Runtime Estimates

⏱ Hands-On Analyst Time

  • Setup + sanity checks (G01–G02): 15–30 min
  • Ordinal 4/6-class fits + confirm outputs: 30–60 min
  • ZIP 4/6-class fits + confirm outputs: 20–40 min
  • Edge cases + log documentation: 30–60 min
  • Package results (logs + screenshots): 10–20 min

🖥 Compute Wall-Clock Time

  • Ordinal 4-class (qpoints=40): ~10–45 min
  • Ordinal 6-class: ~20–90 min
  • ZIP 4-class (qpoints=1): ~2–10 min
  • ZIP 6-class: ~5–25 min
  • Total range: ~40 min – 2.5 hrs (always run K=4 first)
